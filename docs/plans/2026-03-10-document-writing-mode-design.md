## 文档 Agent 写文档模式设计（v1）

日期：2026-03-10

### 1. 目标与非目标

- **目标**
  - 将当前「写文档」能力从简单模板调用，升级为一个有清晰步骤和边界的写作模式。
  - 支持多种文档类型（技术设计 / 教学文档 / 会议纪要 / 周报 / 决策记录等），但保持统一的交互流程。
  - 充分利用当前笔记与跨笔记搜索结果，生成结构清晰、风格可控的 Markdown 文档。
- **非目标（本次不做）**
  - 不实现复杂工作流（任务创建、会议安排等）。
  - 不设计超重模板系统，只保留少量可选的「骨架模板」。
  - 不改动多 Agent 架构，只在现有 Document Agent 上增加「写文档模式」的行为规范。

### 2. 模式触发与整体流程

当 Document Agent 识别到用户意图是「写 / 改一份文档」时，进入写文档模式。无论入口指令如何（“帮我写个技术文档”“帮我整理这次会议纪要”等），都统一走下面的对话流程：

1. **意图确认 / 模式切换**
   - 模型判断当前需求是否为「文档写作」，若不确定，可以用一句简短问题确认：
     - 例：「听起来你是在写一份文档而不是普通问答，我可以帮你按文档来整理，可以吗？」
2. **文档类型确认（必经步骤，不直接套模板）**
   - 通过一个多选问题确认文档类型，例如：
     - 技术设计 / 技术说明
     - 教学 / 教程
     - 会议纪要 / 讨论总结
     - 周报 / 日报 / 进展汇报
     - 决策记录 / ADR
     - 其他（用户自由描述）
   - 如果模型已有强烈猜测，可给出推测 + 选项，让用户确认。
3. **写作 profile 询问（风格 / 受众 / 长度）**
   - 用 2–3 个简短问题锁定写作风格，贯穿整篇文档：
     - 读者是谁？（工程师 / 产品 / 业务 / 外部客户 / 初学者等）
     - 语气？（正式 / 中性 / 轻松）
     - 粗略长度？（短说明 / 1–2 页 / 更长）
4. **生成 2–3 套大纲候选，而不是直接正文**
   - Agent 基于「文档类型 + profile + 当前笔记 + 相关笔记搜索结果」，生成 2–3 套大纲：
     - 技术文档：背景 → 问题 → 约束 → 方案 → 取舍 → 风险 → 后续工作 等。
     - 教学文档：前置知识 → 目标 → 步骤 → 示例 → 练习 → 常见错误 等。
   - 明确告知用户：「先选或调整大纲，再展开正文」。
5. **大纲确认 / 微调循环**
   - 用户可以选择某一版本，也可以要求在某版本基础上增删章节。
   - Agent 只调整大纲，不急于生成正文。
6. **逐节写作（默认一节一节写）**
   - 默认策略：按确认后的大纲，逐节生成正文。
   - 每一节结束时，简短询问是否需要调整语气或详略。
   - 用户也可以显式要求「先一次性写完整文档」，Agent 可给出全量初稿，然后支持按节重写。
7. **可选：模板骨架分支**
   - 对于「会议纪要 / 周报 / ADR 等结构强约束场景」，Agent 可以额外询问：
     - 「你是想要一个结构化的 Markdown 骨架（你自己填），还是我直接帮你写完整草稿？」
   - 只有当用户选择「骨架模板」时，才会触发底层模板工具（见第 3 节）。

### 3. 工具使用策略（read_note / search_notes / draft_document）

#### 3.1 read_note（当前笔记）

- **定位**：写作主上下文的第一来源。
- **策略**
  - 在写文档模式中，Agent 几乎总是应当先调用一次 `read_note`，将当前笔记内容显式纳入思考。
  - 技术文档、会议纪要、决策记录等场景下，优先从当前笔记抽取：
    - 背景 / 目标
    - 关键结论 / 决策
    - TODO / 行动项

#### 3.2 search_notes（跨笔记搜索）

- **定位**：补充上下文，而非强制步骤。
- **推荐触发条件**
  - 用户提到「之前的讨论 / 上次会议 / 之前版本」等历史内容。
  - 当前笔记信息明显不足以支撑完整文档（例如只是一堆零散要点）。
- **使用方式**
  - 将搜索结果作为「可引用素材」，用于丰富大纲与正文的背景和参考。
  - 明确避免直接整段照抄，必要时用引用或总结形式呈现。

#### 3.3 draft_document（模板骨架工具）

- **定位变化**：从「默认写文档路径」降级为**「显式请求模板时的辅助工具」**。
- **触发条件**
  - 用户明确要求模板/骨架，例如：
    - 「给我一个会议纪要的空模板，我自己填。」
    - 「先给我一个技术文档的 Markdown 结构，后面我慢慢补。」
  - 或在对话中用户选择了「只要骨架，不需要完整草稿」。
- **行为约束**
  - draft_document 输出应被视为「可编辑骨架」，而不是最终成品。
  - Agent 在回复中需要解释清楚：「这是骨架，你可以直接在其中填内容；如果需要，我也可以在此基础上帮你写完整版本。」
  - 后续将扩展模板 key，使其覆盖更多文档类型（技术设计、教程、周报、ADR 等），但所有模板都仅作为结构参考。

### 4. 文档类型与写作 profile

#### 4.1 初始文档类型

文档 Agent 在写文档模式下，支持以下类型（后续可扩展）：

- 技术设计 / 技术说明
- 教学 / 教程
- 会议纪要 / 讨论总结
- 周报 / 日报 / 进展汇报
- 决策记录 / ADR
- 其他（用户自由描述，Agent 根据描述选择最接近的内部写作策略）

这些类型主要影响：

- 大纲模板（章节结构）
- 写作重点（例如技术文档更关注约束与取舍，教程更强调步骤和示例）

#### 4.2 写作 profile

每次写文档对话中，Agent 应通过少量问题收集以下信息，并在内部保持一致：

- **受众**：工程师 / 产品 / 业务 / 外部客户 / 初学者 / 混合人群。
- **语气**：正式 / 中性 / 轻松 / 对话式。
- **详略程度**：短说明 / 标准长度（1–2 页） / 更长。
- **语言偏好**：中文 / 英文 / 中英混合（如有必要）。

这些 profile 字段可在后续实现中抽象为配置或上下文对象，供 prompt 与写作逻辑复用。

### 5. 错误处理与兜底策略

- **误判意图**
  - 如果模型将普通问答误判为写文档，Agent 应提供简单的纠正路径：
    - 「我可以直接回答你的问题，也可以帮你整理成一份文档，你更想要哪种？」
- **用户不愿选择类型**
  - 若用户多次拒绝明确文档类型，Agent 可采用温和兜底：
    - 默认使用「技术/通用说明文档」结构，并在回复中说明当前假设。
- **中途切换需求**
  - 用户在大纲阶段改口（例如从技术说明改为教程），Agent 应允许重新生成大纲，而不是在错误类型上硬写。

### 6. 与现有代码结构的关系（高层）

- `src/agents/document-agent/prompts.ts`
  - 将在后续实现中，扩展 `DOCUMENT_AGENT_SYSTEM`，加入「写文档模式」的流程说明与行为约束。
- `src/agents/document-agent/tools.ts`
  - 保持 `read_note` / `search_notes` / `draft_document` 三个工具，但调整调用逻辑：
    - 写文档模式下优先自动使用 `read_note`。
    - 条件触发 `search_notes`。
    - 仅在用户显式要求模板时使用 `draft_document`。
- `src/agents/conversation.ts` 等上层对话逻辑
  - 后续实现时，需要在对话状态中识别「当前是否处于写文档模式」，并将对应信息（文档类型 / 写作 profile）注入 prompt。
