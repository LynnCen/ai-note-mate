# 前端页面体验增强 — 设计文档

> 在现有 AI 笔记应用基础上，分阶段增强交互反馈、编辑器样式、视觉动效与信息结构。阶段顺序由设计推荐，已与用户确认采用方案 1。

---

## 1. 范围与阶段顺序

| 阶段 | 主题 | 说明 |
|------|------|------|
| **Phase 1** | 交互与反馈 | Toast 替代 alert、AI 请求 loading、错误提示、删除确认 |
| **Phase 2** | 编辑器样式与控件 | MDEditor 工具栏/边框/高度、暗色统一、与 shadcn 一致 |
| **Phase 3** | 视觉与动效 | 列表卡片、空状态、加载骨架、过渡动画 |
| **Phase 4** | 信息与结构 | 列表搜索/排序、面包屑、更新时间、字数等 |

**原则**：YAGNI，每阶段可独立交付、可感知提升。

---

## 2. Phase 1：交互与反馈

### 2.1 目标

- 所有异步操作有明确反馈；错误不用 `alert`；破坏性操作有确认。

### 2.2 具体项

- **Toast 系统**
  - 引入 shadcn **Sonner** 或 **Toast**，全局挂载在 layout。
  - **成功 Toast 仅用于不可逆/低频操作**：例如「笔记已删除」。**不对**标题/内容自动保存、AI 接受后保存打 success toast，避免打扰。
  - 错误：API 非 2xx、网络异常、AI 流失败 → Toast 报错（可带简短文案），不再 `alert()`。

- **AI 请求 loading**
  - 点击「AI 处理」或选区浮层某一动作后，在请求期间：
    - 方案 A（推荐）：在 AiResultModal 内显示「生成中…」+ 简单骨架或 spinner，流式内容到达后逐步替换。
    - 方案 B：禁用页头「AI 处理」按钮 + 浮层按钮，并显示 loading 态（如 spinner），流开始后打开 Modal。
  - 采用方案 A：Modal 立即打开，内容区先 loading，首 chunk 到达后开始流式展示。

- **错误提示**
  - 所有 `alert(err)` 改为 Toast error，文案沿用现有中文（如「AI 处理请求失败」「未返回流」）。
  - 可选：网络错误与 5xx 区分文案（如「网络异常，请重试」/「服务暂时不可用」）。

- **删除确认**
  - 点击「删除笔记」后先弹出 **确认对话框**（shadcn AlertDialog）：标题「删除笔记」，描述「确定要删除这篇笔记吗？此操作不可恢复。」，取消 / 确定；确定后再发 DELETE 请求，成功 Toast + 跳转列表。

### 2.3 数据流与依赖

- Toast：layout 或根组件挂载 `<Toaster />`；各页通过调用 `toast.success()` / `toast.error()` 等触发。
- 删除确认：详情页内 state 控制 AlertDialog 的 open；无新 API。

## 3. Phase 2：编辑器样式与控件

### 3.1 目标

- MDEditor 与整站 shadcn/Tailwind 风格统一；工具栏、边框、高度、暗色一致；控件易用不突兀。

### 3.2 具体项

- **容器与边框**
  - 编辑器外框与 shadcn 的 `border-border`、`rounded-lg` 一致；与详情页标题输入框视觉层级一致。

- **工具栏**
  - 使用 MDEditor 自带工具栏；通过 CSS 覆盖背景、边框、按钮 hover/active，使其与 `bg-muted/50`、`text-foreground` 等 token 一致。
  - 可选：工具栏高度或按钮间距微调，避免过密或过矮。

- **编辑区**
  - 字体、行高、内边距与现有阅读体验一致；`data-color-mode` 已与 html.dark 同步，确保暗色下对比度足够。

- **高度与滚动**
  - 保持当前 `height={200}` 或按需微调；`enableScroll={true}` 保留；若需要可增加 `minHeight` 避免折叠过小。

### 3.3 技术要点

- 仅通过 CSS 覆盖 MDEditor 的类（如 `.w-md-editor`、`.w-md-editor-toolbar`），不改其 DOM 结构；暗色由现有 `data-color-mode` 驱动。

## 4. Phase 3：视觉与动效

### 4.1 目标

- 列表与空状态更清晰、友好；加载有骨架或占位；关键过渡有轻微动效，不喧宾夺主。

### 4.2 具体项

- **列表卡片**
  - 保留现有列表布局；可选：卡片 hover 时轻微阴影或 scale，或边框高亮（用 Tailwind transition）。

- **空状态**
  - 首页「暂无笔记」：可加插图或图标 + 简短文案 + 突出「新建笔记」按钮；风格与 shadcn 中性色一致。

- **加载态**
  - 首页：若 `fetchNotes` 未返回前显示骨架列表（3–5 条占位）或统一 loading 文案。
  - 详情页：已有「加载中…」；可改为骨架（标题条 + 编辑区占位）或保持简洁文案。

- **过渡**
  - 路由切换、Modal 开合、AlertDialog 开合：使用 shadcn 自带 transition 或 Tailwind `transition`；列表项出现可选 `animate-in` 轻量动画。

### 4.3 技术要点

- 动效控制在 200–300ms，避免冗长；优先使用 CSS 与现有 shadcn/tw-animate。

---

## 5. Phase 4：信息与结构

### 5.1 目标

- 列表更易扫描与查找；详情页上下文更清晰。

### 5.2 具体项

- **列表**
  - 可选：按更新时间或创建时间排序（默认最近更新在前）；可选：列表顶部搜索框，按标题或内容摘要过滤。

- **详情页**
  - 可选：面包屑「笔记 / 无标题」（或笔记标题）；可选：显示「上次保存」时间或字数统计（仅前端计算）。

### 5.3 技术要点

- 搜索与排序可先做前端（基于当前 store 的 notes）；若以后笔记量增大再考虑服务端搜索。

---

## 6. 实施顺序（与依赖）

1. **Phase 1**：Toast（Sonner）+ AI Modal 内 loading + 错误 Toast 化 + 删除 AlertDialog。无阻塞依赖，可先做。
2. **Phase 2**：在 Phase 1 之后，仅改样式与 CSS，与 Phase 1 无强依赖。
3. **Phase 3**：可与 Phase 2 并行或在其后；依赖现有列表/详情结构。
4. **Phase 4**：在列表与详情稳定后做，依赖 Phase 1 的 Toast/反馈习惯已建立。

---

## 7. 成功标准（简要）

- Phase 1：无 `alert`；删除前有确认；AI 请求有 loading 与错误 Toast。
- Phase 2：编辑器视觉与页内其他 shadcn 组件一致，暗色正常。
- Phase 3：空状态与加载态明确；过渡不刺眼。
- Phase 4：列表可排序或可搜索；详情可选显示辅助信息。

确认后进入实现计划与 Phase 1 开发。
